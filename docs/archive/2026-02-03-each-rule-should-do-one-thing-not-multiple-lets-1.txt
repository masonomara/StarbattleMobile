  The Problem

  This file does five distinct things to find composites:

  1. Reserved Area Exclusions (lines 1383-1448) — when forced regions exactly
  cover a line's need
  2. Adjacent Line Composites (lines 1309-1370) — combining 2-4 adjacent
  rows/columns
  3. Undercounting Composites (lines 127-203) — M rows with N regions, M > N
  4. Overcounting Composites (lines 212-330) — N regions in M rows, N > M
  5. Combination Composites (lines 335-482) — adjacent regions merged (singles,
  pairs, triplets)
  6. Partitioned Regions (lines 489-523) — disconnected components within a
  region

  That's six finders feeding into one analyzer. The analyzer itself has two
  paths:
  - Tiling-based analysis (findAllMinimalTilings)
  - Direct enumeration (enumerateValidPlacements)

  Water's Verdict

  The essence is singular: tight virtual regions reveal forced moves.

  But the implementation is a delta with many tributaries. Each finder is its
  own logic, its own edge cases, its own bugs to maintain.

  If refactoring: the finders could be separate rules that each produce
  composites, feeding a single "tight composite analyzer" rule. The analysis
  logic is already shared — it's the finding that sprawls.
